# analog-guest

This is the messaging engine for Analog — a guest recognition platform for independent cafes, bakeries, and restaurants. This repo handles inbound and outbound messages between venues and their guests, plus the AI generation, classification, and routing logic. It also hosts the internal Command Center (`app/admin/*`) — the Analog-staff debugging surface served on `admin.theanalog.company`. A separate venue-operator dashboard (`analog-operator`) is planned but not yet built; bearer-token auth scaffolding for that future repo lives in `lib/auth/` (see "Auth boundary" below).

## Product principles (do not violate)

- **Recognition, not loyalty.** Guests do not "earn" things — they get recognized. Avoid loyalty-program language ("points," "rewards," "tier," "earn") in any guest-facing output, system prompts, or operator-facing UI copy.
- **Speaker framing is per-venue.** Messages to guests can come from a few different framings depending on what the venue approves: from the venue itself (default), from a named person at the venue ("[Name] from [Venue]"), or from the owner directly when the owner approves that framing. The default is venue-level until told otherwise per venue.
- **Every venue is its own isolated block.** Own messaging number, own voice corpus, own config, own data. Cross-venue queries are Analog-internal only and never exposed to operators.
- **The product is messaging-only.** There is no guest app. Do not assume a frontend for guests. The guest's only surface is the iMessage/SMS conversation with the venue.
- **Voice is the product.** The agent's response style, register, and judgment are what we sell. Treat voice quality as a first-class concern, not a polish-later detail.

## Tech stack

- **Language:** TypeScript, strict mode
- **App framework:** Next.js (App Router) deployed on Vercel
- **Database:** Postgres with pgvector, plus Auth and Storage (Supabase as the current provider)
- **Messaging:** iMessage + SMS fallback (Sendblue as the current provider). One dedicated number per venue. Inbound webhook: `webhooks.theanalog.company/api/webhooks/sendblue` (configured Sendblue-side; route handler at `app/api/webhooks/sendblue/route.ts`). Future webhook integrations (Square, Stripe, Toast) inherit this subdomain convention.
- **LLM:** Anthropic API via the Vercel AI SDK abstraction layer (model-agnostic from day one)
- **Embeddings:** Voyage
- **Drive integration:** Google Drive + Sheets API for the venue onboarding artifact pipeline (06-09 files described below)
- **Analytics:** PostHog for product/event analytics, Slack for operational alerts

Treat the database, messaging, and LLM providers as swappable. Code should depend on internal interfaces (e.g., `lib/messaging/send.ts`), not on vendor names.

## Code conventions

- Industry-standard TypeScript and Next.js naming. PascalCase for components and types, camelCase for functions and variables, kebab-case for filenames, SCREAMING_SNAKE for env vars.
- Prefer functions over classes
- Async/await over `.then()`
- Zod for runtime validation at all API boundaries
- No `any` types. Use `unknown` and narrow.
- Errors are values for internal functions: return `{ ok: true, data }` or `{ ok: false, error }`. Throw only at outer boundaries (route handlers, scripts). Domain-specific aliases formalize the contract: `RAGResult<T>` (`lib/rag/types.ts`), `AIResult<T>` (`lib/ai/types.ts`), `RequireAdminResult<T>` (`lib/auth/`). New helpers should reuse one of these or define a parallel alias rather than inlining the discriminated union.
- Imports use the `@/*` alias for repo-relative paths in app/lib code. Scripts may use either `@/*` or relative imports; see "Module split for testability" for when relative imports are required.

### Module split for testability

Vitest resolves `@/*` aliases via `vitest.config.ts` (THE-231). The remaining failure mode is **module-load-time SDK init** — importing a module that constructs a Voyage / Supabase admin client at the top level will run that init in the test process, even with `vi.mock` (mocks intercept resolution, not transitive eager init). When this happens, split the helper module:

- `scripts/onboarding/<name>-pure.ts` — pure helpers, no `@/*` imports. Tests import from here.
- `scripts/onboarding/<name>.ts` — DB-touching code. Re-exports everything from `-pure` so the CLI can keep a single import.

Example: `ingest-response-review-pure.ts` (parsing, classification, dedupe) + `ingest-response-review.ts` (Supabase upserts, Voyage embed). The CLI imports from the latter; tests import from the former.

### LLM call patterns

- Use Vercel AI SDK's `generateObject` for structured outputs with a Zod schema.
- **THE-157 gotcha: never use `.min()` or `.max()` on number fields in LLM-output Zod schemas.** Anthropic's structured output rejects these. Use `.refine()` or post-LLM validation instead.
- Default model for production agent work: `claude-sonnet-4-6`. Default temperature for generation: 0.7 (variation in phrasing) or 0.3 (idempotent extraction).
- All LLM-output schemas should have a `prompt_version` string field on the wrapper object so we can distinguish output from different prompt iterations.

## Workflow rules for Claude Code

These are not aspirations. They are hard rules. Following them produces good work; skipping them produces drift.

### Linear-first

All work starts from a Linear ticket. If a request arrives without a ticket ID, ask for one before planning. Drive-by changes without an audit trail are not allowed.

### Audit-first

Before writing any code for a new ticket, read:
1. This file (CLAUDE.md)
2. The relevant existing files the new code will sit next to (e.g., for a new script in `scripts/`, read at least one existing script in the same directory)
3. The migrations that touch the relevant tables
4. Any existing tests for the modules being modified

Cite specific file paths and existing patterns in your plan. Don't infer architecture from filenames; read the actual code.

**High-stakes changes require explicit human review.** Any change touching auth (`lib/auth/`, `verifyAnalogAdminAccess`, RLS policies), payment/Stripe code, Sendblue webhook handlers, the agent runtime contract (voice fidelity floor, retrieval floors, universal voice rules R1–R7), or migrations against `messages` / `engagement_events` / `voice_corpus` cannot proceed via `[NEEDS-INPUT]` clarifying questions. Post a `[HUMAN-REVIEW-REQUIRED]` comment on the Linear ticket and stop. Do not propose alternatives, do not partial-implement. Wait.

### Plan → review → build → review → commit

1. **Plan only on the first request.** Output a written plan covering scope, file paths, function decomposition, sequence of operations, existing patterns to reuse, edge cases, what you intentionally chose NOT to do, and open questions. Stop and wait for review.
2. **Build only after the plan is approved.** A second prompt will explicitly authorize the build per the approved plan.
3. **Verify and report.** After building, run `npx tsc --noEmit` and `npx vitest run`. Report file changes, test count, deviations from the plan, and anything you'd push back on.
4. **Commit only after review.** Don't commit until the human says go.

### Comment protocol

**Authorship prefix.** Every Linear comment posted via the MCP starts with `**[FROM CLAUDE CODE]**` on its own line at the top. Linear shows Jaipal as the author of all MCP-posted comments, so this prefix is the only way to distinguish CC comments from chat comments (which use `**[FROM CLAUDE CHAT]**`). Applies to every comment — audits, plans, self-reviews, NEEDS-INPUT flags. Going forward; not retroactive.

**Purpose tag.** When a clarifying question surfaces during work, follow the authorship prefix with `[NEEDS-INPUT]` and each question numbered. Set ticket status to "Awaiting Input" and stop. When resuming, look for the most recent `[DESIGN-ANSWERS]` comment for the user's response. For high-stakes changes (per "Audit-first" above), use `[HUMAN-REVIEW-REQUIRED]` instead — never `[NEEDS-INPUT]`.

**`/work-ticket` auto-polling.** Inside the `/work-ticket` flow, posting a comment that needs a response (plan post, `[NEEDS-INPUT]`) does not exit the session. The command is idempotent and uses `ScheduleWakeup` to auto-resume on a 60s → 300s exponential backoff with a 2-hour cumulative timeout (26 iterations). When the operator replies in Linear, the next wakeup detects the reply and applies a 3-way classification: **Proceed** (advance to next phase), **Modify** (integrate revisions and re-post plan, keep polling), or **Wind-down** (post `[POLLING-CLOSED]` and exit cleanly). Chit-chat / holding-pattern replies post `[POLLING-ACK]` and re-schedule. Terminal states that exit immediately with no wakeup: `[HUMAN-REVIEW-REQUIRED]`, `[POLLING-TIMEOUT]`, `[POLLING-CLOSED]`, and PR-link comments at Phase 5. The operator types `/work-ticket TAC-XXX` exactly once; subsequent harness re-fires of the same prompt are transparent. The agent never auto-closes the ticket — operators close manually (the permission hook enforces this). **Stale-session safety:** each session writes a `[POLLING-STATE]` comment to the ticket and updates it on every wakeup. If a manual re-invocation finds a fresher `[POLLING-STATE]` than the one it would write, it exits immediately to avoid two wakeup chains for the same ticket. This means manually re-running `/work-ticket TAC-XXX` while a session is already polling will appear to "skip" iterations — that's intentional. Outside `/work-ticket`, the default "stop after `[NEEDS-INPUT]`" behavior above still applies.

### Never just acknowledge

If asked to remember, forget, or update something about the user's context, use the `memory_user_edits` tool. Don't reply conversationally with "I'll remember that" — that's a lie if you don't actually update memory.

## Database migrations

Migrations live in `db/migrations/` numbered sequentially. Each is hand-written SQL. Workflow:

1. Write the migration file.
2. Operator applies it via Supabase Studio's SQL editor.
3. Operator runs `npm run db:types` to regenerate `db/types.ts`.
4. If the migration adds a column the script needs to typecheck against immediately, hand-patch `db/types.ts` in the same commit as the script. The next `db:types` run will overwrite the patch with canonical output. Document the patch in the commit message so it's not lost.

### Ordering for backwards-incompatible migrations

When a migration drops a column, renames a column, or adds a `NOT NULL` constraint that the currently-deployed code doesn't yet handle, **deploy the code first, then apply the migration in Studio.** The reverse order opens a downtime window — between Studio-apply and Vercel-deploy, the deployed code queries a schema that no longer exists and every affected request 500s.

**Backwards-incompatible → code first, then schema:**

1. Open PR with the new code (and the migration file checked in).
2. Merge → Vercel deploys the new code. The new code may reference columns that don't exist yet — `tsc` and any runtime path that hits the missing column will fail cleanly (compile error / "column does not exist") rather than producing 500s on a live user-facing path.
3. Apply migration in Studio. Run `npm run db:types`.

**Backwards-compatible (additive only) → order doesn't matter.** A migration that only adds nullable columns, only widens a `CHECK` constraint, or only creates a new index can be applied before or after the code deploy without a downtime window.

**Heuristic:** if `git diff` shows a `DROP COLUMN`, `DROP FUNCTION`, `RENAME COLUMN`, `ALTER ... SET NOT NULL`, or a tightened `CHECK`, treat the migration as backwards-incompatible. Apply Studio AFTER merge — never before.

Source: TAC-272 (2026-05-16). Migration 021 dropped `operators.auth_user_id` while the deployed code still queried it; `admin.theanalog.company` 500'd for the ~3-minute window between Studio-apply and PR merge.

## Phase 5 onboarding pipeline

Phase 5 is the voice-quality cycle that completes venue onboarding. Five files in the venue's Drive folder, numbered 04-08:

- `04-{slug}-menu` (gsheet) — menu CSV, owner-editable
- `05-{slug}-transcript` — onboarding interview transcript
- `06-{slug}-venue-spec-draft.md` — structured venue spec extracted from transcript + airtable + menu. **Central voice brain for the venue.** Original extraction at top; Phase 5 review additions appended in dated subsections at the bottom (always append, never replace). Regenerating only happens if the spec genuinely needs to change — Phase 5 voice training does NOT trigger a re-seed.
- `07-{slug}-test-scenarios.json` — venue-tailored test scenarios (44+ rows for mock-central-perk). Generated by `extract-test-scenarios` from the 06-spec + the categories fixture.
- `08-{slug}-response-review` (gsheet) — populated by `run-test-scenarios`. Owner reviews collaboratively during the Phase 5 meeting, marks `verdict` (approve/edit), provides `edited_message` for rejections, adds `rule:`-prefixed comments for anti-patterns. Then `ingest-response-review` reads it back and updates the corpus + persona surgically.

The pipeline is idempotent at every step. Re-running any script is safe.

### File-naming convention

The 04-09 numbering is meaningful and reserved. Don't introduce other 04-09 files in venue folders. Each script reads its expected files by prefix (`04-`, `06-`, `07-`, `08-`) — adding ambiguous matches will trip the existence guard.

The `-draft` suffix on the 06-file is misleading post-Phase-5 (the file is canonical at that point, not a draft). Slated for rename in THE-185 — bundle with the next touch of any of the three pipeline scripts that reference the filename.

## Drive integration

Google Drive auth uses Application Default Credentials (ADC) via `gcloud auth application-default login`. The OAuth client lives in the analog Google Cloud project; client JSON is at `~/.config/analog/oauth-client.json` (or wherever the operator's local setup put it). Scopes: `openid`, `userinfo.email`, `cloud-platform`, `drive`.

Workspace org policy blocks the stock gcloud OAuth client. Use the project-owned OAuth client (Desktop app type) instead — Workspace's third-party policy doesn't apply to your own org's apps. Pass the client JSON via `--client-id-file` to `gcloud auth application-default login`.

Drive helpers are in `scripts/onboarding/drive.ts`:

- `getDrive()` — auth + Drive client
- `findVenueFolder(drive, parentFolderId, slug)` — locates the venue's folder by slug
- `listVenueFiles(drive, folderId)` — lists files in a venue folder
- `findByPrefix(files, prefix)` — finds the unique file with a given prefix; throws on ambiguity
- `readDriveFileAsText(drive, fileId)` — reads a Drive file as text. For gsheets, exports as CSV.
- `writeMarkdownFile(drive, folderId, name, body)` — writes/upserts a markdown file by name
- `writeJsonFile(drive, folderId, name, obj)` — writes/upserts a JSON file by name (pretty-printed)
- `writeSheetFile(drive, folderId, name, csvBody)` — writes/upserts a native Google Sheet from CSV body. Uses `mimeType: 'application/vnd.google-apps.spreadsheet'` + `media.mimeType: 'text/csv'` so Drive auto-converts on upload. The update path explicitly pins both mimetypes to preserve sharing.

### ADC token refresh (when Drive scripts fail)

When `run-test-scenarios`, `extract-venue-spec`, `seed-supabase`, `extract-test-scenarios`, or `ingest-response-review` fails with `Request had insufficient authentication scopes`, ADC needs reauth with explicit Drive scope:

```bash
gcloud auth application-default login \
  --client-id-file='/path/to/client_secret_*.apps.googleusercontent.com.json' \
  --scopes='openid,https://www.googleapis.com/auth/userinfo.email,https://www.googleapis.com/auth/cloud-platform,https://www.googleapis.com/auth/drive'
```

**Why this is needed:** the default scopes for `gcloud auth application-default login` (with no `--scopes` flag) do NOT include `drive`. The bare command authenticates successfully but every Drive API call returns `Request had insufficient authentication scopes`. The `--scopes` flag is mandatory for any script that touches Drive.

**Why the `--client-id-file`:** Workspace org policy blocks the stock gcloud OAuth client. The project-owned OAuth client (Desktop app type, `client_secret_*.apps.googleusercontent.com.json`) bypasses that policy because Workspace's third-party policy doesn't apply to your own org's apps.

## Observability and alerting

Three layers, each filling a different gap. Don't conflate them.

- **PostHog events.** Product/event analytics. Captures inbound/outbound events, fidelity scores, classification outputs, etc. Configured to fire to Slack on certain event filters. Today only fires on errors; THE-187 expands coverage to silent failures (low fidelity, empty corpus, classification confidence drops, latency spikes, regeneration cycles, webhook silence). PostHog event emission is wrapped in try/catch — failures must not crash the agent path.
- **Slack alerts** (THE-159). Diagnostic-rich alert content for fired alerts. Narrow scope: improves what existing alerts say, doesn't change what fires.
- **Trace-level observability.** Per-run visibility into the agent's reasoning (corpus retrieval scores, prompt content, fidelity self-rating, etc.). Wired up via Langfuse Cloud (THE-200). `lib/observability/langfuse.ts` is a thin wrapper exposing `startAgentTrace` returning an `AgentTrace` (`span()`, `update()`, `flushAsync()`); spans expose `span()` / `generation()` / `update()` / `end()`. Wrapper invariants:
  - Never throws. SDK errors are caught at the wrapper boundary and logged via `console.warn` — observability is diagnostic, not load-bearing.
  - No-op fallback when `NODE_ENV=test`, `LANGFUSE_ENABLED=false`, any of the three env vars (`LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`, host) are missing, or SDK init throws. The host can come from `LANGFUSE_BASE_URL` (preferred — matches the SDK's `baseUrl` constructor field) or `LANGFUSE_HOST` (legacy alias accepted for compatibility with Langfuse's own docs / Vercel integration; slated for removal post-pilot). `BASE_URL` wins when both are set. In no-op mode `trace.id === ''`, all methods are silent no-ops, and agent code runs unchanged.
  - Trace ID is available synchronously the moment `startAgentTrace` returns. `schedule-and-send.ts` writes `trace.id || null` to `messages.langfuse_trace_id` at insert time so traces are queryable from the row.
  - Span tree per agent run: `agent.inbound` (or `agent.followup`) → `context_build` → `classify` (inbound only) → `retrieve` → `retrieve_knowledge` (conditional, gated by `shouldRetrieveKnowledge` in `lib/agent/stages.ts`: always for inbound, only for followup `event` / `manual` triggers) → `generate` → `send`. The `generate` span has post-hoc `generate.attempt_N` sub-spans synthesized from `attemptScores` — these have no real per-attempt timing; threading the trace into `lib/ai`'s regen loop for true timing is tracked in THE-215. Knowledge retrieval degrades gracefully on Voyage / DB error (logs + returns `[]`, span closes normally with `matchCount=0`); voice retrieval continues to fail closed because voice failure breaks voice fidelity itself.
  - Spans carry **metadata always** (counts, scores, IDs in `output` / span input) and **content gated** by `LANGFUSE_CAPTURE_CONTENT` (full inbound body, full retrieved corpus chunks, full system + user prompts, per-attempt generations, full sent body — folded into `output.content` when on, dropped entirely when off; THE-216). Default-on; set `LANGFUSE_CAPTURE_CONTENT=false` to revert to THE-200's metadata-only shape (e.g. for a venue with strict data-handling requirements). The flag is read once at module init — flipping it requires a redeploy. `lib/agent/trace-content.ts` centralizes the per-stage content shape (`buildRecognitionContent`, `buildCorpusContent`, `buildGenerateContent`, `buildGenerateAttemptContent`); `trace.captureContent` is exposed so handlers can skip building heavy payloads when no one will look at them. Coverage gap doc lives in the THE-216 PR description.
  - `flushAsync` is awaited in the `finally` block of each handler. Both handlers run inside a `waitUntil` keep-alive window from their callers (webhook route, cron) so the flush completes before the function returns.
  - `/admin/health` reports four states via `app/admin/(authed)/health/check-langfuse.ts`: **Active** (all keys + known host, green), **Disabled** (`LANGFUSE_ENABLED=false` set explicitly, neutral), **Misconfigured** (some keys present but not all, or unrecognized host — red, with the specific reason in the detail line), **Not configured** (no `LANGFUSE_*` vars set, neutral — local-dev / no-op is appropriate). The check does not actively probe Langfuse — the SDK has no synchronous ping, and probing on every health-page load would pollute the trace stream.

## AI agent runtime contract

These constraints are real and have caused bugs:

- **Voice fidelity floor (3-tier, TAC-212):** the agent self-rates voice fidelity. Below `SEND_FIDELITY_FLOOR` (0.4) → refused with red alert (existing behavior — `generateStage` returns `'refused'`, no row is persisted, no send). At or above 0.4 but below `AUTO_SEND_FIDELITY_FLOOR` (0.6) → queued for operator review (`applyApprovalPolicyStage` fires the `fidelity_below_auto_send_floor` trigger). At or above 0.6 → continues to the resource-commitment + sticky-pending gates inside `applyApprovalPolicyStage`. Both floors live in `lib/agent/stages.ts`. The intermediate `VOICE_FIDELITY_LOW_THRESHOLD` (0.5) still fires the `voice_fidelity_low` PostHog observation event for the 0.4–0.5 band (now always queued).
- **Corpus retrieval floor:** retrieval is fail-closed *on the inbound path*. Need at least `MIN_STRONG_MATCHES` (1) chunk scoring at or above `STRONG_MATCH_SIMILARITY` (0.3); below that, the agent bails to a fallback acknowledgment. Followup path skips the gate entirely (THE-231) — operator-initiated sends shouldn't be blocked by sparse corpus retrieval. Defined in `lib/agent/stages.ts`.
- **Classifier confidence routing (TAC-240, v1.11.0):** 3-tier handling in `classifyStage`. Above `CLASSIFICATION_CONFIDENCE_LOW_THRESHOLD` (0.7) → keep the classifier's pick, no event. Between `CLASSIFICATION_CONFIDENCE_REROUTE_THRESHOLD` (0.3) and 0.7 → keep the pick + fire `classification_low_confidence` (operator should check). Below 0.3 → rewrite the returned `Classification.category` to `'unknown'` so the agent ships a holding response; original category and confidence are preserved on the PostHog event payload (`autoRoutedToUnknown=true`) and on the returned `Classification` for observability. Regen path (`lib/voices/regenerate-with-critique.ts`) does not apply the reroute — operator iterates on the output. Classifier inputs were also widened to receive `recentMessages` + `guestState` so one-word inbounds ("yes", "iced") can be classified in conversational + relationship context; classifier `temperature` dropped from the SDK default 1.0 to 0.2 (analytical task); inbound is truncated to `MAX_CLASSIFIER_INPUT_CHARS` (1000) before classification while generation still gets the full body.
- **Em-dash hard-block (THE-225):** R3 in `SYSTEM_TEMPLATE` says no em dashes, but Sonnet still emits them occasionally even at high voice fidelity. After regen attempts exhaust, a regex hard-blocks remaining `—` and rewrites the offending text; voice fidelity is recomputed on the final shipped body, not the original LLM output. Lives in `lib/ai/generate-message.ts`.
- **Visit history block (TAC-234, v1.13.0):** `RuntimeContext.recentVisits` carries up to `MAX_VISIT_HISTORY_TRANSACTIONS` (20) transactions within the past `MAX_VISIT_HISTORY_DAYS` (90) for the guest, most-recent-first. Rendered as `## Visit history` in the user prompt with bracketed time-delta bullets matching the `## Recent conversation` style. Replaces the v1.12 single-transaction `## Last visit` block. The intro line tells Sonnet to use this for pattern recognition (recommendations, "what haven't I tried") — not to recite history back at the guest. Skipped at the block level for `welcome` and `opt_out`. Empty `recentVisits` omits the block entirely (parallels `## Recent conversation`'s skip-on-empty; distinct from `## Venue knowledge`'s render-explicit-no-match — that pattern exists because the model might invent venue facts, R9 is the safety net there). Projection lives in `lib/agent/extract-recent-visits.ts`.
- **Guest relationship surfacing (TAC-234, v1.13.0):** `recognition.state` is now rendered as `Guest relationship: <state>` directly after the inbound framing line, symmetric with the classifier's TAC-240 surfacing. Closes the tone-calibration gap between regulars and new guests. Single source of truth: the recognition snapshot — surfacing only, no recomputation.
- **runtimeToProse field-presence rendering (TAC-234, v1.13.0):** the per-category switch in `lib/ai/prompts/serializers.ts` was replaced with three independent conditional blocks: inbound framing (one consistent `The guest just sent: "..."` line for any inbound), `perk_unlock` data, `event_invite` data. Carve-outs ("just asked", "(opt-out request)") collapsed into the unified inbound line — the category-specific instructions in the system prompt already convey question vs statement intent. The mutual exclusion between inbound and outbound runtime contexts is enforced by the orchestrator entry points (`handleInbound` populates `inboundMessage`; `handleFollowup` populates `perkBeingUnlocked` or `eventBeingInvited`); type-system enforcement is tracked separately as TAC-243 (backlog).
- **Recent conversation block:** the last 14 days of messages between this guest and this venue are loaded into RuntimeContext. THE-173 added this; future changes to the window should consider impact on synthetic-guest seeding (`run-test-scenarios` seeds messages >30 days old to keep this block empty during testing).
- **Today's date in context:** `RuntimeContext.todayInVenueTimezone` is set from `new Date()` at runtime, formatted in the venue's timezone. THE-174 added this so the agent answers "are you open today?" correctly.
- **Universal voice rules R1-R7:** live in `SYSTEM_TEMPLATE`. Apply to every agent on every venue. Don't reference actions the guest didn't take, give today's specific answer, no em dashes, don't reference physical artifacts the agent doesn't have, don't redirect to alt channels for things the venue can answer, yes/no questions get yes/no answers, don't restate context already in the conversation.
- **Mechanic eligibility (THE-170):** mechanics with `min_state` set are filtered out of `RuntimeContext.mechanics` when the guest's recognition state is below the gate. Mechanics with active redemptions are also filtered (`one_time` policy blocks forever after any `mechanic_redeemed` event; `renewable` blocks within `redemption_window_days`). Helpers: `isStateAtLeast` and `filterEligibleMechanics` in `lib/recognition/`. The filter runs at load time in `build-runtime-context.ts` after `recognition.state` is computed, sharing `computedAt` so the filter timestamp matches the recognition snapshot. Empty list is meaningful — the serializer renders an explicit "do not offer perks of any kind" instruction when `mechanics: []`. Empty-list framing is load-bearing for closing the eligibility-leak failure case where Sonnet would otherwise improvise mechanic offers from voice corpus matches.
- **Voice vs knowledge retrieval:** the agent retrieves from two corpora. `voice_corpus` (always-on, fail-closed) supplies HOW the venue speaks — style exemplars rendered as `## Examples of how the venue actually communicates`. `knowledge_corpus` (gated by `shouldRetrieveKnowledge`, fail-graceful) supplies WHAT IS TRUE about the venue — narrative content (sourcing stories, staff personalities, mechanic explanations, philosophy) rendered as `## Venue knowledge` with `[primary: ...]` and `[secondary: ...]` lines per chunk. Knowledge fires for every inbound and for followup triggers `event` / `manual`; skips `day_*` cron triggers. Failure asymmetry is intentional — voice failure breaks voice fidelity (the whole point of the message), knowledge failure just means a less specific reply, still in the venue's voice. Knowledge retrieval uses `match_knowledge_corpus` (migrations 013 + 017), top-K=4, with a `KNOWLEDGE_CONFIDENCE_FLOOR_DEFAULT` of 0.7 (matches the classifier low-confidence threshold for symmetry).
- **Knowledge corpus confidence floor (TAC-242):** `KNOWLEDGE_CONFIDENCE_FLOOR_DEFAULT = 0.7` is applied as the default `min_confidence` on every `match_knowledge_corpus` call. Pre-TAC-242 retrieval passed no floor (admitted all chunks); the new floor retroactively excludes any `confidence_score < 0.7` row from the prompt. Production seed defaults are `0.85` (direct quotes) and `0.85` (synthesized) so this is a no-op for normally-seeded venues, but hand-loaded low-confidence rows will silently disappear from retrieval. Operators can override per-call via `RetrieveKnowledgeContextInput.minConfidence` (passing `0` is honored).
- **Guest context block (TAC-296, v1.15.0):** `RuntimeContext.guestContext` carries the parsed per-guest accumulating context (dietary, home base, life events, observations) loaded from `guests.context` JSONB. Rendered as `## Guest context` in the user prompt by `formatGuestContext` (`lib/ai/prompts/serializers.ts`), inserted **between Visit history and Recent conversation** so reading order is "who they are → what they can get → what they've recently bought → what we know about them → what was recently said." No category gate — useful for every category including `welcome` (e.g., NFC-tap from a known phone whose context says "vegan"). Empty context omits the block entirely (zero tokens). Hard char budget ≈ 500 tokens; serializer truncates observations to 5 floor first, then drops oldest life_context entries if still over. Parser is permissive: `filterActiveLifeContext` drops expired + malformed `expires_at` entries per-entry (mirrors THE-150's `filterActiveContext`), `toParsedGuestContext` truncates observations to `OBSERVATION_RENDER_LIMIT`=10 at read time. The write path lives on `GeneratedMessageSchema.contextUpdate` (required object, inner `structured` + `observation` both optional — the no-op shape is `{}`); dispatched by `handle-inbound.ts` + `handle-followup.ts` **between `generateStage` success and `applyApprovalPolicyStage`** so context capture reflects what the agent UNDERSTOOD from the inbound regardless of whether the draft ships, queues, or refuses. Arrays in `structured` are REPLACED, not appended (per ticket spec — the prompt instruction tells the model to include any existing entries it wants to keep when emitting a life_context patch). `observation` shortcut appends to `observations[]` with runtime-stamped `captured_at`. Identity-column sync (`first_name` / `last_name`) happens inside the same UPDATE statement (multi-SET, row-atomic). Last-write-wins concurrency — two near-simultaneous inbounds from the same guest can clobber each other's deep-merge; documented in "Common gotchas." Hallucination guardrail is the prompt rule + a Langfuse `context_write` span tag — no refusal regex, no fidelity self-check.
- **Knowledge corpus tag-aware retrieval (TAC-242, v1.12.0):** knowledge_corpus rows carry a two-bucket tag model. `primary_tags` is a closed enum (`sourcing`, `staff`, `mechanic`, `menu`, `philosophy`, `recommendations`, `events`, `history`, `space`, `policies`, `logistics`, `other` — namespacing allowed: `staff_phoebe`, `mechanic_<slug>`) used for retrieval routing; `secondary_tags` is free-form descriptive context that surfaces in the prompt but doesn't affect routing. Validated at the parse boundary via `isCanonicalPrimaryTag` in `lib/schemas/knowledge-tags.ts` — fail-loud on non-canonical primary tags so extraction drift surfaces immediately. The category → primary tag preference map (`lib/agent/knowledge-tag-mapping.ts`) sets routing for four categories: `mechanic_request` and `perk_inquiry` → `mechanic`; `recommendation_request` → `recommendations` / `menu` / `sourcing`; `event_question` → `events`. Other categories use cosine-only retrieval. When a category's preference yields zero matches (sparse corpus for that topic), `retrieveKnowledgeStage` falls back to a no-filter query so recall doesn't collapse. The same pattern is mirrored in `lib/voices/regenerate-with-critique.ts` for the regen path. The `## Venue knowledge` block now ALWAYS renders when retrieval ran (even on empty results — explicit "no specific venue knowledge matched" framing makes R9 fire reliably); the block is omitted only when `knowledgeChunks` is `undefined` (retrieval was gated off). The legacy `tags` column on `knowledge_corpus` stays for one cycle and is dropped in a follow-up migration. Adding a category to the routing map requires an explicit code change to `CATEGORY_TO_PRIMARY_TAG_PREFERENCE`.

The hard guardrail against leaking internal taxonomy (`post-visit day 7`, `cold lapsed`, `perk just unlocked`) to the guest lives in the `follow_up` category instruction (`lib/ai/prompts/categories/follow-up.ts`), not the block — the block uses positive medium-framing, the instruction carries the never-enumerate rule. **Inbound-XOR-outbound invariant** enforced by `throw`-on-violation entry-point assertions in `handleInbound` + `handleFollowup` (loud, dev-time-catching; top-level catch routes to `fireRedAlert` with stage='context_build') — same posture as TAC-234, no dependency on the TAC-243 discriminated-union refactor. No category gate (the follow_up-category instruction's pointer covers the natural caller; other categories never see this field set in practice).
- **Guest commitments block (TAC-297, v1.16.0; commitment-id render TAC-302, v1.18.0; emission discipline TAC-302 follow-up, v1.19.0):** `RuntimeContext.activeCommitments` carries the open + pending_ack rows for this guest at this venue (projected through `toActiveCommitment` in `lib/schemas/guest-commitment.ts`). Rendered as `## Active commitments` in the user prompt by `formatActiveCommitments` (`lib/ai/prompts/serializers.ts`), inserted **between Guest context and Recent conversation** so reading order is "...what we know about them as a person → what we've promised → what was recently said." Per-commitment line shape: `- [type] description (id: <uuid>, code: XXXX, status: ...) — promised <delta>` (code omitted for `recommendation`; id always rendered). Tail copy frames the block as PERMISSION to ask about arrival timing if it fits, NOT a standing directive (TAC-297 design call #5 — interrogation-risk mitigation). Empty list omits the block entirely (zero tokens). No category gate. The write path lives on `GeneratedMessageSchema.commitment` and `GeneratedMessageSchema.arrivalCapture` (both required objects with all-optional inner fields — the no-op shapes are `{}`, mirrors TAC-296 `contextUpdate`); helpers `isEmptyCommitmentEmission` + `isEmptyArrivalCapture` short-circuit before any DB hit. **The commitment path is post-send (not pre-gate)** — intent rides the queue via `messages.pending_commitment` jsonb and materializes only on successful dispatch (TAC-297 design call #1). **The arrival-capture path IS pre-gate** — fires regardless of whether the draft ships/queues/refuses, mirroring the TAC-296 `contextUpdate` precedent (what the agent UNDERSTOOD is independent of what we SAID back). `dispatchArrivalCapture` in `lib/agent/dispatch-arrival-capture.ts` is the shared helper called from both `handle-inbound.ts` (live push) and `handle-followup.ts` (defensive no-op). **TAC-302 (v1.18.0):** the `id:` segment is the verbatim value the model copies into `arrivalCapture.referencesCommitmentId`. Pre-TAC-302 the system prompt told the model to populate that field "from the ## Active commitments block" but the block didn't render the id — every arrival signal either no-op'd (id omitted) or hallucinated against the 4-char `code:` (CAS rowcount=0). Result: no commitment ever reached `pending_ack`, the imminent push never fired, the morning-of cron had nothing due. The system-template `# Arrival capture` block AND the serializer intro both carry the "id is system-internal, never read aloud, never include in your reply to the guest" framing — that's the primary line of defense against UUID leakage into conversational text (R1 / R7 don't directly cover structured-emission discipline). **TAC-302 follow-up (v1.19.0):** UAT after v1.18.0 confirmed the id-render fix landed (model's `content.reasoning` explicitly names the commitment by code), but surfaced the next layer — the model narrates the correct arrival action inside reasoning prose, then emits `arrivalCapture: {}` (no-op shape) because it talks itself out of the field with reasons like "the heads-up was already asked last turn, no need to repeat." The fix reframes the `# Arrival capture` block from action-based ("when to emit") to detection-based ("populate whenever ACTIVE COMMITMENTS non-empty AND inbound contains arrival intent — including confirmations, closers, previously-discussed times"), explicitly decouples the structured emission from the conversational heads-up ask ("detection, not communication"), enumerates four specific anti-patterns Sonnet uses to suppress emission and tags each `IRRELEVANT` / `A CONFIRMATION IS A SIGNAL` / `END-OF-CONVERSATION IS WHEN ARRIVAL DETECTION MATTERS MOST` / `DOESN'T MATTER`, and adds a worked example matching the prod failure trace (prior-turn heads-up ask → current-turn confirmation "ok i'll come in tomorrow around 8" → MUST emit `signal: "scheduled"`). No schema change — the schema is structurally permissive (no-op shape is valid by design per Anthropic structured-output reliability) and the lever is the prompt instruction. v1.17.0 (TAC-300) was a `GuestContextPatchSchema` slim reshape with no runtime-contract block change; documented under "Common gotchas → `GeneratedMessageSchema` optional-field budget."

### Approval policy gates (TAC-212, v1.14.0)

`applyApprovalPolicyStage` (`lib/agent/stages.ts`) runs immediately after a successful `generateStage` on both the inbound and followup orchestrators. It returns `{ action: 'send' }` or `{ action: 'queue', triggers, primaryTrigger, compMatchedPattern, existingPendingDraftId }`. On `queue`, the orchestrator calls `persistOrRegenQueuedDraft` (`lib/agent/schedule-and-send.ts`) — same row shape as `scheduleAndSend` via the shared `buildOutboundInsert` helper, but with `status='pending_review'`, `review_state='pending'` (TAC-258 migration 018 predicate), `review_reason=<primaryTrigger>`, `sent_at=null`, `provider_message_id=null`. No Sendblue dispatch. The `AgentResult.queued` discriminated-union member carries `outboundMessageId`, `triggers`, and `primaryTrigger`.

**Five triggers compose; any one queues.** Enumeration order (the order the gate evaluates and the order they appear in `triggers[]`):
1. `fidelity_below_auto_send_floor` — `voiceFidelity` in `[0.4, 0.6)`. (`< 0.4` is already refused upstream.)
2. `model_flagged` — `GeneratedMessageSchema.requiresOperatorApproval=true`. Reason is in `GenerateMessageResult.approvalReason`. Two prompt triggers teach the model when to flag: the universal comp rule in `# Resource commitment self-flag` (`SYSTEM_TEMPLATE`), and per-mechanic instructions appended to flagged-mechanic bullets in `formatMechanicEligibility`.
3. `comp_regex_backstop` — `matchComp` (`lib/agent/comp-backstop.ts`) hits a high-precision operator-action regex. Runs **independently** of `model_flagged` — when the model misses a self-flag on a comp, the regex still queues. False positives are acceptable; false negatives are the failure mode this exists to prevent.
4. `previous_pending_held` — `findPendingDraft` lookup against the migration 018 partial index finds an existing `review_state='pending'` row for the (venue, guest) pair. **Fails OPEN** on DB error: lookup throws or returns an error → returns `null` → gate proceeds to send. Migration 020 (TAC-264) is the structural backstop against concurrent rapid-inbound races that slip past this fail-open read.
5. `commitment_type_gated` (TAC-297) — `GeneratedMessageSchema.commitment.type ∈ {comp, hold, discount}` AND `description` non-empty. Fires regardless of `requiresOperatorApproval` self-flag — the structured emission IS the backstop (TAC-297 design call #2). Recommendation type does NOT fire this trigger. **The comp regex stays as belt-and-suspenders** for prose-comp-without-emission cases the model might produce.

`primaryTrigger` is **not** `triggers[0]`. It's picked via `PRIMARY_TRIGGER_PRIORITY` (severity-of-signal order): `commitment_type_gated` > `comp_regex_backstop` > `model_flagged` > `previous_pending_held` > `fidelity_below_auto_send_floor`. `primaryTrigger` lands on `messages.review_reason` (the field the operator queue UI surfaces first), so the most-severe signal is what gets shown. **Commitment_type_gated tops the priority** because it carries explicit type information (the operator queue UI sees "hold" / "comp" / "discount" directly rather than just "regex matched some comp prose").

**Follow Up button bypass:** `handle-followup.ts` skips `applyApprovalPolicyStage` entirely when `input.trigger.reason === 'manual'`. The Command Center Follow Up button is the only path that emits `reason='manual'`; operator already approved by clicking. Same trigger-based heuristic the existing `skipHumanFeelDelay` wiring uses. Cron-triggered followups (`day_*` / `event`) run the gate. Manual followups also bypass pending-detection — the partial unique index from migration 020 doesn't block them because they write `review_state='auto_sent'`, not `'pending'`. A pending card + an operator-sent manual outbound can legitimately coexist on the same guest.

**Generation schema fields:** `GeneratedMessageSchema` (`lib/ai/generate-message.ts`) rigidly requires `requiresOperatorApproval: boolean` and `approvalReason: string` — no `.optional()` because the Anthropic structured-output validator is more reliable with explicit presence. `approvalReason` defaults to `""` when `requiresOperatorApproval=false`. Both are threaded onto `GenerateMessageAttempt` and `GenerateMessageResult` for trace observability.

**Per-mechanic flag (`mechanics.requires_operator_approval`):** added in migration 019, defaults `false`. The eligible-mechanics serializer (`formatMechanicEligibility` in `lib/ai/prompts/serializers.ts`) annotates flagged mechanic bullets with `[operator approval required: …]` so the model knows committing this mechanic should set `requiresOperatorApproval=true`. Loaded by `build-runtime-context.ts` through `EligibilityCandidate` → `EligibleMechanic`. SQL UPDATE template for toggling per-venue is in Common gotchas.

**PostHog events:** `captureDraftQueued` (`lib/analytics/posthog.ts`) fires whenever the gate queues a NEW pending draft (INSERT path), with `triggers` + `primaryTrigger` + `voiceFidelity` + `modelRequiresApproval` + `modelApprovalReason` + `compRegexMatchedPattern` + `hasPreviousPending` + `kind` ('inbound' | 'followup') + `category` + bodies. `captureDraftRegenerated` (TAC-264) fires when the gate queues a REGENERATION of an existing pending row (UPDATE path), with the same shape plus `originalDraftId` (the `messages.id` that was updated in place) and `priorReviewReason` (the row's `review_reason` BEFORE overwrite — captured pre-UPDATE so analytics can surface trigger evolution across regens). Both Slack relay on (pilot ops needs visibility into both paths). Mirrors the `voice_fidelity_low` / `dash_violation_persisted` shape.

**Demo guest bypass (TAC-284).** When `ctx.guest.isDemo === true` (the `guests.is_demo` flag, migration 022), `applyApprovalPolicyStage` still evaluates all four triggers — so the would-have-queued set is accurate — then **short-circuits to `{ action: 'send', reason: 'demo_bypass' }` regardless of trigger outcome.** The bypass is total: every trigger, **including the comp regex backstop**, is overridden. The comp backstop is therefore **not a guarantee for demo guests** — a demo guest who is committed a comp gets the comp auto-sent with no operator review. This is intentional (demo guests are teammates' own phones, not real customers); the visibility guarantee is the `demo_bypassed_approval_gate` PostHog event, which fires whenever the would-have-queued trigger set is non-empty and **Slack-relays only when `comp_regex_backstop` is among those triggers** (the irreversible-financial-commitment case). The bypass is **fail-closed**: only the literal boolean `true` bypasses — `undefined` / `null` / a missing column flow through the normal policy. Both orchestrators also pass `skipHumanFeelDelay: ctx.guest.isDemo` to `scheduleAndSend`, and the demo send is stamped `messages.review_reason='demo_bypass'` (the auto-send path otherwise leaves `review_reason` null) so the row is self-describing without a PostHog cross-reference. Manual followups (`reason='manual'`) bypass the gate entirely and never carry the `demo_bypass` stamp — a manual outbound is an explicit operator send, not a bypass. Flip a guest's flag via the SQL template in Common gotchas; the flag is read fresh every agent run (no runtime caching).

### Single pending draft per guest invariant (TAC-264, v1.14.0)

**The invariant:** at most one row with `review_state='pending'` per `(venue_id, guest_id)`. Enforced at the storage layer by the partial unique index `idx_messages_one_pending_per_guest` (migration 020); enforced at the application layer by `persistOrRegenQueuedDraft` (`lib/agent/schedule-and-send.ts`) which branches on `existingPendingDraftId` to UPDATE in place vs INSERT.

**Two persist modes via `persistOrRegenQueuedDraft(ctx, generation, primaryTrigger, existingPendingDraftId)`** — returns `{ outboundMessageId, action: 'inserted' | 'updated', priorReviewReason }`:

1. **INSERT** — `existingPendingDraftId === null` AND the partial unique index doesn't fire. Status quo INSERT via `buildOutboundInsert` with the queue-path overrides. `priorReviewReason` is always `null` on this path.
2. **UPDATE in place (regenerate)** — `existingPendingDraftId !== null`. Conditional UPDATE gated on `id = existingPendingDraftId AND review_state = 'pending'` so a TOCTOU race vs. `dispatchOperatorOutbound` produces rowcount=0 rather than corrupting the operator's just-dispatched body. Captures `priorReviewReason` via a SELECT before the UPDATE so analytics can observe trigger evolution.

**Columns UPDATED on regen** (the "current state of the draft" subset): `body`, `voice_fidelity`, `prompt_version`, `category`, `reply_to_message_id`, `langfuse_trace_id`, `review_reason`. `updated_at` is auto-bumped by `trg_messages_updated_at`.

**Columns PRESERVED on regen** (operator-visible history + queue ordering): `status='pending_review'`, `review_state='pending'` (unchanged — **no demotion to send across regen**), `created_at` (FIFO queue position — operator's "this guest has been waiting since X" mental model), `last_operator_action_at`, `last_operator_id`, `previous_review_state`, `id`, `venue_id`, `guest_id`, `direction`, `generated_by`, `response_review`.

**Race recovery (bounded loop, `RACE_RECOVERY_MAX_ATTEMPTS=3`)**:
- INSERT path: a concurrent inbound for the same (venue, guest) wins the unique-index race; the losing INSERT receives `code='23505'`. We call `findOpenPendingRow` (local helper inside schedule-and-send.ts, mirrors the read shape of `findPendingDraft` in stages.ts) to re-fetch the now-existing row and recurse into UPDATE on the next loop tick. If the racing row vanished between violation and read (operator immediately dispatched), the loop retries INSERT.
- UPDATE path: a TOCTOU race vs. `dispatchOperatorOutbound` clears the pending slot between our `findPendingDraft` + our UPDATE. Either the prior-reason SELECT returns no row OR the conditional UPDATE returns rowcount=0 — either way we drop `existingPendingDraftId` and recurse into a fresh INSERT.
- Sustained ping-pong (3 ticks without convergence) fires a red alert (`stage='persist'`) and throws. Caller maps to `AgentResult.failed`.

**No-demotion invariant**: structurally enforced by the `previous_pending_held` trigger inside `applyApprovalPolicyStage`. When `findPendingDraft` succeeds and returns a row, the trigger fires unconditionally; `triggers.length > 0`; the gate returns `action='queue'`. A high-fidelity untriggered regenerated draft on a guest with an existing pending row still queues — never sends out from under the operator. The fail-open path (`findPendingDraft` errors → returns null → trigger doesn't fire → action might be 'send') is the only loophole; migration 020's partial unique index closes it at the storage layer.

**Manual followup interaction**: out of scope for this invariant. `handle-followup.ts` bypasses the approval gate AND pending-detection when `trigger.reason === 'manual'` — the operator clicked "send" and expects fast dispatch. The manual outbound writes `review_state='auto_sent'`, doesn't violate the partial unique index, and can legitimately coexist with a pending card on the same guest.

### Follow-up engine (TAC-123, v1.21.0)

Daily per-venue scan that detects which enrolled guests are due for an unprompted touch (post-visit day_N, cold_lapsed re-engagement, perk_unlock), runs Gate 1 (`canSendFollowup`) + the claim-before-dispatch idempotency machinery, and hands off to the existing `handleFollowup` → `applyApprovalPolicyStage` (Gate 2) → dispatch tail. **No fork** of the outbound path. Concrete sibling to TAC-297's commitments processor; both live in their own folders (`lib/followups/` / `lib/guests/commitments-due.ts`) and the shared seam extraction is deferred per design call #3.

**Cron + trigger.** Hourly GH Actions tick (`.github/workflows/followups-due-cron.yml`) hits `/api/cron/followups-due`. The route delegates to `processDueFollowups(now)` in `lib/followups/engine.ts`. Per-venue local-hour filter (against `followup_rules.cron_hour_local`, default 10) lives in the processor; the cron fires hourly UTC and the processor matches per venue (Vercel Hobby blocks sub-daily Vercel cron — same posture as commitments-due).

**Detectors → reasons[] aggregation.** Three pure detectors in `lib/followups/detectors.ts`:
- `detectPostVisitReason(lastVisitAt, messagingCadence, now)` — highest ENABLED day_N tier the guest has crossed since their last visit. "Highest only" — skipped tiers don't backfill (sending day_1 + day_3 + day_7 at once reads as spam).
- `detectColdLapsedReason(lastVisitAt, currentState, rules, now)` — last visit older than `absence_window_days` AND current state in `lapsed_eligible_states` (default `['regular','raving_fan']` — we don't try to re-engage guests who never engaged).
- `detectPerkUnlockReason({currentState, eligibleMechanics, announcedMechanicIds, rules})` — first eligible mechanic the guest can now unlock that hasn't been announced yet (`announcedMechanicIds` parsed from `followup_log.dedup_key` rows with `perk:` prefix).
The engine aggregates hits into a `FollowupReason[]`, picks the primary via `PRIMARY_REASON_PRIORITY` (`perk_unlock > cold_lapsed > post_visit_day_14 > … > post_visit_day_1`), maps the primary to `FollowupTrigger.reason` (renaming `post_visit_day_N` → `day_N`; `cold_lapsed` / `perk_unlock` pass through), and puts the rest in `trigger.additionalReasons`. For perk_unlock, the chosen mechanic also rides as `trigger.perkMechanic` — `buildAiRuntime` reads it and populates `perkBeingUnlocked` on the AI runtime (first production caller of that AI-runtime field).

**Gate 1 — `canSendFollowup` (`lib/agent/followup-rules.ts`).** Pure function. Suppression order: opt-out → quiet hours (venue-local with midnight-crossing handled; default 21:00–08:00) → recent conversation (`recent_conversation_hours`, default 48) → weekly cap (rolling 7d, default 1; counts engine-initiated only — manual sends from TAC-249 don't count) → time-bound per-reason dedup (`cold_lapsed` only — `cold_dedup_days`, default 30). Intent-bound dedup (don't re-fire post_visit_day_7 for the same `last_visit_at`; don't re-fire perk_unlock for the same mechanic) is NOT here — that's the claim layer's job via the `followup_log.dedup_key` UNIQUE constraint.

**Gate 2 — `applyApprovalPolicyStage` (existing, TAC-212).** Engine sends are subject to all the same triggers — voice fidelity floor, model self-flag, comp regex backstop, sticky pending, commitment-type gate. A perk-bearing followup that requires operator approval queues normally; the operator queue UI surfaces it with whatever `primaryTrigger` the gate picked.

**Idempotency — claim-before-side-effect (TAC-123 plan-review).** Beats double-send under overlapping cron runs. Per guest, per run:
1. Engine builds `FollowupClaimRow[]` from detected reasons + their dedup_keys (`day_N:<last_visit_at_iso>` / `cold:<last_visit_at_iso>` / `perk:<mechanic_id>` per `dedupKeyForReason`).
2. `claimFollowupLogRows` atomically INSERTs all rows. UNIQUE conflict on `(venue_id, guest_id, dedup_key)` → entire INSERT rolls back (atomic), helper returns `{conflict: true}`, engine skips this guest.
3. `handleFollowup` dispatches. AgentResult `sent` / `queued` → `finalizeFollowupLogClaim` stamps `message_id`. `refused` / `failed` → `releaseFollowupLogClaim` DELETEs the claim so dedup isn't burned (next morning tick can re-attempt).

**Manual followups bypass the engine entirely.** TAC-249's Command Center button calls `handleFollowup` with `trigger.reason='manual'`; that path skips Gate 2 (`applyApprovalPolicyStage`) AND skips Gate 1 (`canSendFollowup` lives in the engine, not the orchestrator). Manual sends don't write `followup_log` rows and don't count toward `weekly_cap`.

**`messages.category` on engine sends.** All engine sends persist as `category='follow_up'` regardless of which reasons fire (TAC-123 plan-review call). The perk-weaving clause lives in `lib/ai/prompts/categories/follow-up.ts` and only fires when category is `follow_up`; routing perk-bearing engine runs to `category='perk_unlock'` would mean the clause never applied. The existing `perk_unlock` MessageCategory remains for the inbound / standalone perk moment (different tuning, untouched). `triggerToCategory` in `lib/agent/handle-followup.ts` maps both day_* and `perk_unlock` and `cold_lapsed` to `'follow_up'`.

**Per-venue config — `venue_configs.followup_rules` (migration 028).** Zod schema in `lib/schemas/followup-rules.ts`; default in `FOLLOWUP_RULES_DEFAULT`. Fields: per-reason `enabled` toggles, `absence_window_days`, `lapsed_eligible_states`, `cold_dedup_days`, `weekly_cap`, `recent_conversation_hours`, `quiet_hours_start_local` / `quiet_hours_end_local`, `cron_hour_local`. `parseFollowupRules` is fail-OPEN — null/missing/malformed jsonb returns defaults with a `console.warn` (same posture as `filterActiveContext`). The pre-existing `messaging_cadence` column stays the pure day-schedule (`{day_1, day_3, day_7, day_14}`); the engine reads both.

**Observability.** Two PostHog events in `lib/analytics/posthog.ts`: `followup_suppressed` (per-guest, fires from canSendFollowup blocks; PostHog + Slack relay so pilot ops can chart unexpected spikes) and `followup_scan_complete` (once per processor run, PostHog only; aggregated summary + per-venue breakdown). Both events are leaf — failures swallowed via `capturePostHogEvent`.

**Tunables** are surfaced read-only at `/admin/tunables` via `lib/tunables/manifest.ts` — five `agent_runtime`-category entries (`followup_absence_window_days`, `followup_cold_dedup_days`, `followup_lapsed_eligible_states`, `followup_weekly_cap`, `followup_recent_conversation_hours`) and three `timing`-category entries (`followup_quiet_hours_start_local`, `followup_quiet_hours_end_local`, `followup_cron_hour_local`). All sourced from `FOLLOWUP_RULES_DEFAULT` — runtime reads `venue_configs.followup_rules` per venue and the viewer shows the canonical default.

## Auth boundary (bearer-token path for analog-operator)

The venue-operator path is bearer-token authenticated via `verifyOperatorRequest` (or its `getCurrentOperator` wrapper, or the `withOperatorAuth(handler)` HOF for route handlers). The helper resolves the Supabase auth JWT to our internal `operatorId` plus an `allowedVenueIds` array. Route handlers do venue-mismatch checks themselves (`allowedVenueIds.includes(targetVenueId)`) and choose their response code — by convention `app/api/operator/*` returns 404 (not 403) for out-of-allowlist messages so existence isn't leaked; the helper itself only ever produces 401. RLS is the longer-term defense; the helpers are the v1 enforcement point.

**Status today:** the bearer path is live as of TAC-258 — five route handlers under `app/api/operator/*` (queue / approve / edit / skip / undo) consume `withOperatorAuth`. The `verifyAnalogAdminRequest` variant (bearer + `is_analog_admin = true`) remains forward-scaffolding for the future `analog-operator` admin-only surface. Active admin auth for `app/admin/(authed)/*` uses the cookie-session path via `verifyAnalogAdminAccess(authUserId)`, documented in "Admin scaffold" below. Don't conflate the two — the cookie path is for Analog staff in the Command Center; the bearer path is for venue operators in the mobile app + future dashboard.

### Multi-method auth + linking (TAC-272)

**Schema:** `operators` carries two nullable UUID columns — `auth_user_id_phone` and `auth_user_id_email` — each UNIQUE and FK-by-convention to `auth.users.id`. A CHECK enforces at least one is populated. `phone_number` is NOT NULL going forward. Both the bearer-token path (`verifyOperatorRequest`) and the cookie-session path (`verifyAnalogAdminAccess`) resolve the operator with the same OR-match: `auth_user_id_phone.eq.X OR auth_user_id_email.eq.X` where `X` is the just-verified `auth.uid()`. The RLS policy `operators_select_own` mirrors the OR clause with explicit `IS NOT NULL` guards on each side.

**Linking:** `linkOperatorByAuthUser(authUserId)` in `lib/auth/link-operator.ts` is the single point where an `auth.users` row gets bound to an `operators` row. Flow: `supabase.auth.admin.getUserById` → probe phone-vs-email identity → fast-path OR lookup → match against `operators` by `phone_number = authUserPhoneToE164(auth.users.phone)` OR `email = auth.users.email` → idempotent UPDATE of the appropriate column. Errors-as-values per the `RAGResult` / `AIResult` convention — six failure modes (`auth_user_not_found`, `auth_user_has_no_identity`, `no_matching_operator`, `multiple_matching_operators`, `already_claimed_by_different_user`, `db_error`). Never throws.

**Three call sites for linking:**
- `app/admin/auth/callback/route.ts` (eager) — right after `exchangeCodeForSession`, before the redirect to `/admin`. `no_matching_operator` → redirect with `?error=no_access`; other link failures → `?error=link_failed`.
- `lib/auth/verify-jwt.ts` (lazy) — when the OR-lookup misses, call `linkOperatorByAuthUser` and retry. Self-heals any auth user that authenticated before the callback was wired (and any future "auth-but-no-link" case).
- `lib/auth/verify-analog-admin.ts` (lazy, cookie-session) — same pattern, plus a follow-up `select is_analog_admin where id = operatorId` because the link helper doesn't return the admin flag.

**Phone normalization:** Supabase stores `auth.users.phone` WITHOUT the leading `+` (e.g. `"18777804236"`); `operators.phone_number` stores E.164 WITH `+` (e.g. `"+18777804236"`). `authUserPhoneToE164` in `lib/auth/normalize-phone.ts` bridges them at match time — Supabase manages `auth.users`, so we don't normalize at storage. Idempotent (an input already with `+` returns unchanged), null-safe, whitespace-trimming.

**Studio-only RPC drop (021):** A `link_operator_auth(text, text)` RPC existed in Studio (created during TAC-206) but had zero call sites in app code — orphaned linking infrastructure that never landed a caller. Migration 021 drops it. The TS-side `linkOperatorByAuthUser` is the canonical replacement. **Lesson worth remembering:** when CLAUDE.md flags "some policies may have been added directly in Studio, not via migrations" for RLS, the same risk applies to SQL functions. Audit Studio for both before any migration that touches `operators` (or any other table that's been long-lived enough to accumulate Studio-only objects). **Update (023, TAC-285):** the RPC is re-introduced under the same signature because `analog-operator/lib/auth/operator.ts` calls it post-OTP / magic-link to link the auth user to the operators row. The TS-side helper remains canonical for analog-guest's own surfaces; the RPC is the operator-app surface only. Now versioned via migration 023, not Studio-only.

## Operator API (TAC-258 + TAC-207 + TAC-277 + TAC-297 + TAC-299)

The mobile-operator approval queue is served by nine route handlers under `app/api/operator/*` (SINGULAR — applies to every operator endpoint without exception), on the **guest host** (the middleware host-gate only restricts `/admin/*` — `/api/operator/*` flows through). When a dedicated operator host comes online, the routes can move under a host-rewrite without touching the handlers. Five TAC-258 queue routes (`queue`, `messages/[id]/{approve,edit,skip,undo}`), the TAC-207 `POST /api/operator/devices` (APNs token registration), the TAC-277 `GET /api/operator/messages/[id]/thread` (full conversation history behind the edit screen), the TAC-297 `POST /api/operator/commitments/[id]/acknowledge` (heads-up card swipe-right), and the TAC-299 `POST /api/operator/commitments/[id]/draft-decline` (heads-up card swipe-left → agent-drafted apology persisted as pending, commitment cancelled).

**Path note (TAC-207 surfacing case).** The TAC-207 ticket draft text referenced the plural `/api/operators/devices` and the iOS client (TAC-288) was initially built against that. Server shipped at singular per convention; a hotfix briefly moved the server to plural to match the client during UAT; then both sides realigned on singular as part of TAC-207 cleanup. Lesson worth remembering for future cross-repo work: ticket draft text isn't a contract — confirm the path against the established convention before the client encodes it.

**Auth.** All five TAC-258 routes plus the TAC-207 devices route wrap their handler in `withOperatorAuth` (from `lib/auth/operator-auth.ts`). The HOF runs `verifyOperatorRequest` (bearer-token JWT → `operators.id` + `operator_venues` allowlist), shapes `AuthError` as 401/403 JSON, and threads `{operator, params}` into the inner handler. Operator-venues with **zero rows** means an operator has no venue access — the queue returns `{drafts: []}`, the action endpoints return 404 (don't leak existence).

**Auth-failure body shape — Contract-bound routes bypass the HOF.** `withOperatorAuth` forwards `AuthError.message` verbatim into the 401 body (e.g. `'missing Authorization header'`, `'invalid or expired token: JWT expired'`). That's fine for the six legacy operator routes that predate cross-repo Contracts. **Routes bound to a cross-repo `## Contract` MUST shape the 401 body to match the Contract literal** (e.g. `{ error: 'unauthorized' }`) instead of leaking `err.message`. Pattern: call `verifyOperatorRequest` inline with a `try { ... } catch (err) { if (err instanceof AuthError) return NextResponse.json({ error: 'unauthorized' }, { status: 401 }); throw err }` block. TAC-277's `GET /api/operator/messages/[id]/thread` is the first route to do this — apply the same posture to any future Contract-bound operator endpoint (404 body becomes `'not_found'`, 500 body becomes `'internal_error'`, etc., per the Contract). Don't back-port to the legacy six unless their sibling clients adopt a Contract — touching the HOF in place would change wire shape for all six at once.

**State machine.** `messages.review_state` is a **separate axis from `messages.status`** (migration 018). Five values: `pending` (draft awaiting review — TAC-212 will set this), `approved` (operator approved, Sendblue dispatched), `edited` (operator edited body, Sendblue dispatched with the edit), `skipped` (operator chose nothing), `auto_sent` (never queued — set by the autonomous-agent path via `lib/agent/schedule-and-send.ts`). NULL for inbound rows and pre-TAC-258 outbounds (the migration backfilled the latter to `'auto_sent'`).

**Idempotency.** Approve + edit + skip use optimistic conditional UPDATEs that gate on `review_state='pending'`. Rowcount=0 means another caller already acted (or the row was never pending) — surface as 200 `{status: 'already_acted', reviewState: ...}`. The migration 006 `provider_message_id` unique constraint is the second backstop against double-send. There's no explicit transaction wrapper — the conditional UPDATE is itself atomic, and the small TOCTOU window before the Sendblue call is acceptable v1.

**Undo.** 3-second server-side window enforced via `now() - last_operator_action_at < 3000ms`. Only skip→pending is truly revertible; approve and edit can't be unsent so undo on those fires a PostHog event with `undoneAfterDispatch=true` and leaves state alone. Operator-mismatch (different operator's action) returns 403. Outside the window returns 409 Conflict with `code: 'UNDO_WINDOW_EXPIRED'`. v1 supports single-level undo only.

**Edit path persistence.** Mobile-operator edit overrides `messages.body` (canonical sent text invariant — see plan v3 §7). The operator's text is also written to `response_review.editedMessage` (the "human edited this" signal), and the AI's pre-edit draft is captured in `response_review.originalAiBody` (closes the Langfuse-retention gap so single-row diff queries work). Voice corpus ingestion uses `upsertCorpusEdit` with `source_ref = 'operator-approve:{messageId}'` (registered in `lib/voice-training/channels.ts`) in `'replace'` mode. **Anti-corpus-poisoning rule still applies** — only the operator's final text is embedded; the AI draft never is. The route mirrors cc-review's circuit-breaker chain: corpus → response_review stamp (last). On already-acted, do NOT re-stamp `response_review` (the first operator's stamp wins).

**v1 failure-recovery gap.** If Sendblue throws after the optimistic state flip, the row is left in `review_state='approved'` (or `'edited'`) with `provider_message_id=NULL`. The queue no longer surfaces it; the operator can't retry via the queue. Recovery is manual SQL or a future `failed_dispatch` reconciliation cron. Mitigation: dispatch errors return 502 from the route — the mobile client surfaces a generic "send failed" toast. **Follow-up ticket needed** for `failed_dispatch` state + reconciliation cron; track post-merge.

**PostHog events.** `operator_message_approved`, `operator_message_edited`, `operator_message_skipped`, `operator_message_action_undone` — all in `lib/analytics/posthog.ts`. IDs-only payload shape per the operator-side precedent (`voice_critique_committed`); bodies live on the row and Langfuse. `operator_message_edited` carries length-delta scalars (`bodyLengthBefore`, `bodyLengthAfter`, `bodyLengthDeltaPct`), not the bodies themselves. None of these post to Slack — they're product analytics, not operational alerts.

**Full-thread fetch (TAC-277).** `GET /api/operator/messages/[id]/thread` returns the full conversation history (guest-at-venue) for the operator-app edit screen. Path param `id` is the messageId the operator just tapped on (any message in the thread works as an entry point, not just the pending draft). Auth uses the Contract-bound inline pattern documented above — 401 body is `{ error: 'unauthorized' }` (not `err.message`); 404 body is `{ error: 'not_found' }` for invalid-UUID, message-not-found, AND out-of-allowlist (uniform per Contract — see CLAUDE.md → Auth boundary on existence-leak); 500 body is `{ error: 'internal_error' }`. Response is `{ messages: ThreadMessage[] }` ordered oldest→newest, capped at the most-recent 200 (the helper does `ORDER BY created_at DESC LIMIT 200` server-side and reverses in JS). Delegates to `loadGuestThread` in `lib/operator/thread.ts`. Empty-body messages (reactions, status pings) filtered server-side. No PostHog event — read-only fetch, no operator action. Sibling client: TAC-290.

**APNs push (TAC-207).** `POST /api/operator/devices` accepts `{ token: string, platform?: string }` (`token` Zod-validated: hex-only, 32-256 chars, opaque-length per Apple's guidance; `platform` is sent by the iOS client today as `"ios"` but **not persisted** — iOS-only pilot, no column. Zod's default object behavior strips unknown keys, so `platform` is silently accepted + ignored. If multi-platform support arrives, add a column AND validate here at the same time). Upserts onto `operators.apns_device_token` + `apns_token_updated_at` (migration 024). Re-registration is idempotent. The token is consumed by `sendDraftFlaggedPush` in `lib/notifications/send.ts`, fired fire-and-forget from `handle-inbound.ts` + `handle-followup.ts` (via `waitUntil`) immediately after `persistOrRegenQueuedDraft` succeeds in the queue branch. Trigger filter is **operator-decided** (plan-review §1): fire on `model_flagged` / `comp_regex_backstop` / `fidelity_below_auto_send_floor`; skip on `previous_pending_held` (regen of an already-pushed draft). Two PostHog events: `push.sent` (per send attempt, `ok:boolean` carries success vs. failure — no Slack relay; analyst-facing) and `push.token_invalid` (APNs 410 Gone OR 400 BadDeviceToken — nulls the token columns + Slack-relays because the operator's push pipeline is non-functional until they reinstall the app). Privacy invariant: payload carries `{ aps: { alert: {title, body}, badge, sound }, draftId, guestId, operatorId }` only — no message contents, ever (asserted in `send.test.ts`). Body is `"Reply to {firstName} — {context}"` where context is a categorical trigger label.

**Heads-up commitments (TAC-297 + TAC-299).** The TAC-258 queue endpoint's response gained an additive `commitments: HeadsUpCommitment[]` field (sibling to the existing `drafts: QueueDraft[]`); shape locked at the cross-repo Contract. TAC-297 shipped the base shape `{ id, type, guest: {name}, description, code, expected_arrival, created_at }`. **TAC-299 added two additive fields** consumed by the operator-app heads-up card: `recognitionState: GuestState | null` (drives the recognition pill — typed via the canonical `GuestState` union from `lib/recognition/types.ts` rather than a re-typed local; same value set as `QueueDraft.recognitionState`) and `sourceMessageId: string | null` (the messageId the operator-app routes into `/api/operator/messages/[id]/thread` to render the inbound conversation thread). Powered by `listHeadsUpQueue` in `lib/operator/heads-up-queue.ts` — main query joins `guests.first_name` via PostgREST embedded relations; **TAC-299 added a second batched query** against `guest_states` keyed on the collected `guest_ids` (deduped) filtered to `exited_at IS NULL` (the partial-index `idx_guest_states_current` hot path) AND the `venue_id IN allowedVenueIds` scope. Two queries because PostgREST nested ordering+limit on embedded relations is unreliable for "latest row per parent"; this is O(2) round trips deterministically. Recognition-state lookup degrades gracefully on error (logs + leaves the map empty → all commitments project `recognitionState=null` rather than failing the whole list — the heads-up card still renders, just without the pill). **`POST /api/operator/commitments/[id]/acknowledge`** uses the Contract-bound inline-auth pattern (matches TAC-277): 401 `{error: 'unauthorized'}`, 404 `{error: 'not_found'}` for invalid-UUID + commitment-not-found + out-of-allowlist (uniform per existence-leak rule), 409 `{error: 'already_acknowledged'}` (CAS lost with row visible — already acknowledged / cancelled / never reached pending_ack), 500 `{error: 'internal_error'}`. Delegates to `markAcknowledged` (`lib/guests/commitments.ts`), a CAS-gated UPDATE on `(status='pending_ack' AND venue_id IN allowedVenueIds)`. PostHog event: `operator_commitment_acknowledged` (`venueId`, `guestId`, `commitmentId`, `operatorId`, `timeToActionMs`, `type`) — IDs-only payload shape mirroring TAC-258's draft events. No Slack relay. **Companion arrival-push (`sendCommitmentArrivalPush` in `lib/notifications/send-commitment-push.ts`):** TAC-207-style fanout fired from `handle-inbound.ts` (imminent CAS win → live push, agentRunId threaded) and the hourly cron route `/api/cron/commitments-due` (scheduled fires, agentRunId=null). Payload `{ aps: {alert: {title, body}, badge, sound}, commitmentId, guestId, operatorId }` — content-free (no commitment description, no inbound text). Body is `"{firstName} arriving {context} — {typeLabel}{ code}"`; `code` omitted for `recommendation` type, included for comp/hold/discount. Badge sums pending drafts + pending_ack commitments per operator's allowed venues (single source of truth across both push surfaces). PostHog discriminator: `surface: 'commitment_arrival'` on `push.sent` / `push.token_invalid` events. **Known duplication:** `loadRecipients` + `nullOperatorToken` + badge-count are duplicated in `send-commitment-push.ts` vs `send.ts`. Extraction to a shared `lib/notifications/recipients.ts` is a follow-up — duplicating now to keep TAC-297 scoped.

**Operator-initiated decline (TAC-299).** `POST /api/operator/commitments/[id]/draft-decline` is the swipe-left counterpart to `/acknowledge`. Contract-bound inline-auth: 401 `{error:'unauthorized'}`, 404 `{error:'not_found'}` for invalid-UUID + commitment-not-found + out-of-allowlist (uniform per existence-leak rule), 409 `{error:'invalid_state'}` (commitment exists but is not `pending_ack` — distinct from 404 because the row was found, just in the wrong state — different operator-app toast), 422 `{error:'refused'}` (voice fidelity below `SEND_FIDELITY_FLOOR`), 500/502 `{error:'internal_error'}` (DB load failure / pipeline failure inside `handleOperatorDecline`), 200 `{messageId}`. The route loads the commitment row (allowlist-scoped), invokes `handleOperatorDecline` (`lib/agent/handle-operator-decline.ts` — see the `lib/agent/` paragraph above for the persist-not-send structural invariant + skipped-gate posture), then calls `markCancelled` AFTER the pipeline succeeds (cancellation is **trigger-time**, not send-time — per TAC-299 design decision #1: the operator's swipe-left intent is load-bearing even if they abandon the edit screen; `editAndSend` stays commitment-agnostic). `markCancelled` CAS lost (concurrent ack from another operator) or DB-errored is **logged + accepted**: the decline draft is already persisted; the route returns 200 with `messageId` and the PostHog event records `commitmentCancellationRaceLost=true` for observability. PostHog event: `operator_draft_decline_initiated` (`venueId`, `guestId`, `commitmentId`, `messageId`, `operatorId`, `type`, `timeToActionMs`, `commitmentCancellationRaceLost`). No Slack relay. Single-pending-draft interaction (TAC-299 design decision #2): when the guest already has a pending draft, `persistOrRegenQueuedDraft` routes to UPDATE-in-place via migration 020's partial unique index — the decline **clobbers** the prior pending reply (operator's "we can't fulfill" supersedes the auto-generated reply); `captureDraftRegenerated` PostHog event observes the trigger transition. `operator_decline_initiated` joins the set of `messages.review_reason` values surfaced by the queue UI but is NOT in `APPROVAL_TRIGGERS` (the policy gate doesn't fire it; the route does) — `lib/operator/queue.ts` carries a separate `ExtraReviewReason` union for this exact reason so the label-map type stays exhaustive at compile time. Sibling client: TAC-298 (operator-app swipe-left wiring).

**Commitments cron (TAC-297, morning-of model).** `/api/cron/commitments-due` is hit hourly to fire each venue's day-prep push at its local morning hour, NOT at each commitment's stated `expected_arrival` time. One push per pending_ack commitment per venue at opening — deliberate day-prep accountability (the imminent path covers just-in-time off the inbound).

**Trigger source is GitHub Actions, NOT Vercel cron** — the project stays on Vercel Hobby, whose cron tier caps granularity at daily/weekly. The workflow `.github/workflows/commitments-due-cron.yml` runs on `cron: '0 * * * *'` (UTC) + `workflow_dispatch` (manual smoke-test trigger) and one curl step authenticated by the same `CRON_SECRET` Bearer the route validates. **Two repo settings required:** `PROD_BASE_URL` as a repo variable (the guest-host origin, e.g. `https://webhooks.theanalog.company`) and `CRON_SECRET` as a repo secret (same value as Vercel's `CRON_SECRET` env var). Without both set, the workflow exits 1 loudly rather than 200-and-silent. **Reliability caveat:** GH Actions scheduled runs can lag the target minute under platform load — the morning-of model tolerates ±1h precision; revisit (Vercel Pro native cron, or QStash) if drift bites or when the project moves to a commercial plan. The route handler is auth-equivalent to a Vercel cron call (dev-skip pattern matches webhook-silence). **Vercel `vercel.json` currently lists only the daily `webhook-silence` cron** (Hobby-compatible); anything sub-daily must live on GH Actions until the project upgrades.

**Eligibility per row in `processDueCommitments(now)`** (`lib/guests/commitments-due.ts`):
1. `findScheduledOpenCommitments()` returns every `status='open' AND arrival_signal='scheduled'` row (no time filter — the date filter is timezone-bound and lives below). `imminent`-signal rows are excluded at the SQL boundary; they fire off the inbound, never here.
2. Look up the row's venue timezone. Skip if missing (counts as `invalid`).
3. **Morning-hour gate.** Compute the hour-of-day in the venue's timezone via `Intl.DateTimeFormat`. Skip unless it equals `MORNING_HOUR_LOCAL` (currently `7`, exported constant). Mismatches count as `notMorningHour` — they'll be picked up on a future hourly tick when their venue's local time rolls around to 7am.
4. **Date gate.** Compute `expected_arrival_date_in_venue_tz` and `today_in_venue_tz` (YYYY-MM-DD via `en-CA` locale — lexicographically comparable). Skip if `expected_date > today_date` (`future`). Process if `<=` — **lean catch-up** means a past-due scheduled commitment whose morning run was missed (created after that day's tick, or GH Actions outage) fires on the next morning tick.
5. CAS-transition via `transitionToPendingAck` (gated on `status='open'`); push fires via `waitUntil` only when CAS won. A scheduled commitment that ALSO got an imminent inbound before the morning tick is already `pending_ack` → CAS rowcount=0 → no double-push.

`MORNING_HOUR_LOCAL = 7` is a module constant for v1; per-venue override via `venue_configs` is a follow-up. Summary shape: `{ scanned, transitioned, skipped, notMorningHour, future, invalid, errored, pushed }` — split out so an operational dashboard can distinguish "didn't fire because not morning" from "didn't fire because data integrity" from "didn't fire because CAS lost to a racing imminent inbound."

**Concrete, not generic** (design call #3) — when the follow-up engine lands, it writes its own sibling `processDueFollowups(now)` and we extract the shared "find-eligible → CAS-transition → side-effect" seam THEN.

## Cross-repo contracts

When a ticket spans this repo and `analog-operator` (or any other sibling repo) — TAC-207 ↔ TAC-288 is the canonical example — the ticket description MUST contain a `## Contract` section that locks the boundary character-exact:

- Endpoint paths (full path string, HTTP method)
- Request body shape and a concrete example
- Response shapes per status code (including the 401/404/409 negatives)
- Required env vars with format notes (multi-line PEM vs single-line secret, etc.)

The Contract section is the single source of truth. Both repos implement against it; at integration time the Contract resolves any disagreement.

**Deviation rule.** If implementation needs to diverge from the Contract — for example, the codebase convention is `/api/operator/*` (singular) but the Contract drafted `/api/operators/*` (plural) — the ticket description gets updated FIRST during the plan-review phase. The implementation never silently diverges from a still-stale Contract. The 2026-05-27 TAC-207 incident — server shipped singular without updating TAC-288's plural spec, the iOS client hit 404 on registration, recovery cost a hotfix plus a re-alignment cycle — is the documented case for why this rule exists.

**Sequencing rule.** Cross-repo work ships sequentially, **server-first**. Before the sibling client ticket starts, the server endpoint must be deployed AND verified via `curl` against the Contract's exact request shape, returning the expected status code. The bar is "401 on missing auth" (proves the route exists, parses, and reaches the auth check) — NOT "404 on missing route," NOT "400 on bad body." A 404 means the route wasn't deployed where the Contract said it would be; a 400 on a valid Contract-shaped body means the body schema diverged.

**UAT gate.** Manual end-to-end UAT is a precondition for marking the server ticket Done — never deferred to the sibling client ticket. Unit tests passing on both sides does not prove the contract is consistent. TAC-207 surfaced this: server tests green, client tests green, integration broken on the first push attempt 9 hours after deploy.

## Common gotchas

- **`@/*` aliases in tests.** Resolved via `vitest.config.ts` (THE-231); previously a recurring cause of test-side import failures. Module-split pattern is still required when a transitively-imported module runs heavy SDK init at module load (see "Module split for testability").
- **Zod `.min()`/`.max()` on LLM output number fields.** Anthropic's structured output rejects these. Use `.refine()` or post-LLM validation. (THE-157.)
- **Permissive schema + filter-time validation pattern.** When the same field is validated at two boundaries (input parse + runtime), prefer strict at the offline boundary (parser/seed) and permissive-with-defensive-filter at the live boundary (runtime). Crashes during seed are catchable; crashes during agent runs hurt guests. Three instances: THE-157 (`.min()/.max()` rejected by Anthropic structured output, validate post-LLM), THE-150 (`expiresAt` stored as plain `z.string().optional()`; `filterActiveContext` parses + drops malformed entries with `console.warn` instead of failing the whole `venue_info` JSONB parse), THE-170 (mechanic `min_state` is strict `z.enum(GUEST_STATES)` at the parser boundary inside `parse-venue-spec.ts`, but `isStateAtLeast` at runtime treats unknown values defensively — drop the gated mechanic, log, don't crash).
- **`filterActiveContext` (THE-150).** Pure helper at `lib/schemas/venue-info.ts`. Drops `currentContext` entries whose `expiresAt` is past or equal to `now`; entries with no `expiresAt` are permanent; malformed `expiresAt` logs and drops. Wired in `build-runtime-context.ts` via the same `computedAt` reused for the recognition snapshot (single "now" per agent run).
- **Re-seeding via `npm run seed-venue` for voice training.** Don't. Phase 5 voice training uses surgical updates via `ingest-response-review`. Re-seed only when the venue spec markdown itself changed structurally.
- **Drive ADC re-auth.** Tokens expire. If you see `invalid_grant` / `invalid_rapt`, re-run `gcloud auth application-default login --client-id-file=... --scopes=...`.
- **gcloud Python version.** Workspace install can break against newer Python. If `gcloud` errors with Python version mismatch, set `CLOUDSDK_PYTHON` to a specific Python 3.10-3.14 binary.
- **Migration application order.** Always apply the migration in Supabase Studio BEFORE running scripts that depend on it locally. Hand-patches to `db/types.ts` are temporary; `npm run db:types` is the resolution.
- **Preview-env parity for server-side env vars.** Every `process.env.*` reference that runs server-side must be set on **both** the Production AND Preview Vercel environments. Preview parity has bitten twice on the operator-API surface (`SUPABASE_SECRET_KEY`, then `VOYAGE_API_KEY` — both initially set on Production only). Failure mode is HTTP 500 with **empty body** from the route, because `lib/db/admin.ts` (and similar helpers) throw on missing env at call time and the throw escapes the route handler before any JSON response is shaped — debugging from the client side looks like a code crash. Mitigation, run as part of post-merge hygiene: grep the changed PR diff for `process.env.` references, cross-check each name against the Preview env var list in Vercel project settings, fix any gaps before opening the preview-protected route to QA.
- **APNs env vars fail SILENTLY (TAC-207).** Unlike the Supabase/Voyage case above, missing `APNS_*` vars on Preview don't surface as a 500 — the agent path continues normally (push is fire-and-forget via `waitUntil`, errors-as-values, never throws), and `sendDraftFlaggedPush` returns early with `ok=false / error='env_missing'` PostHog events. **Symptom is "push didn't arrive,"** not "API call broke." Diagnostic chain: (1) PostHog `push.sent` events with `error='env_missing'` / `error='jwt_failed'` — the env var name is in `errorDetail`; (2) check Vercel project settings for `APNS_AUTH_KEY` / `APNS_KEY_ID` / `APNS_TEAM_ID` / `APNS_BUNDLE_ID` / `APNS_ENV` on the failing environment; (3) for `APNS_AUTH_KEY` specifically, the value is the PEM contents (multi-line, includes `-----BEGIN PRIVATE KEY-----` lines) pasted directly into the Vercel value field — a path or a single-line transformation won't parse.
- **`venue_configs.brand_persona` vs `venues.brand_persona`.** It lives on `venue_configs`, not `venues`. Check `build-runtime-context.ts` for the canonical access pattern.
- **`response_review.reviewedVia` is optional with a read-time default (TAC-258).** `MessageReviewSchema` added `reviewedVia: z.enum(['cc_review', 'mobile_operator']).optional()` without bumping `schemaVersion`. Pre-TAC-258 cc-review rows don't carry the field. Readers MUST go through `getReviewedVia(review)` from `lib/schemas/message-review.ts`, which defaults missing values to `'cc_review'` (the only pre-existing channel). New writes on the cc-review path stamp the field explicitly; the mobile-operator edit path always stamps `'mobile_operator'`. Don't try to read `review.reviewedVia` directly — you'll get `undefined` on legacy rows and silently fall through wrong branches.
- **`response_review.originalAiBody` only set on the mobile-operator edit path (TAC-258).** For mobile-operator edits, the route overwrites `messages.body` with the operator's text (preserving the `body = canonical sent text` invariant cc-review depends on), and captures the pre-edit AI draft in `response_review.originalAiBody`. cc-review never sets `originalAiBody` — `messages.body` already carries the AI draft because cc-review is post-hoc. So a non-null `originalAiBody` is the strongest signal of "this row went through the mobile approval queue's edit path."
- **`voiceAntiPatterns` shape (THE-236).** Stored shape is `Array<{ text, source: 'auto' | 'manual', authorOperatorId?, addedAt? }>`. **The DB is in a dual-shape state in perpetuity:** legacy `string[]` entries from rows last written before THE-236 stay legacy until any writer touches them; new and migrated rows persist as struct. `BrandPersonaSchema` accepts both forms and normalizes string entries to `{text, source: 'manual'}` on parse, so all in-process code sees the struct shape uniformly — but the JSONB on disk does not. Three rules follow:
  - **Always go through `BrandPersonaSchema`.** Reading `venue_configs.brand_persona->'voiceAntiPatterns'` directly via SQL or a JSONB path expression breaks on the legacy half of the fleet — `jsonb_array_elements_text` works on legacy rows, `... ->> 'text'` works on migrated rows, neither works on both. If you need a SQL query that touches anti-patterns, write the union explicitly (`CASE WHEN jsonb_typeof(elem) = 'string' THEN elem #>> '{}' ELSE elem ->> 'text' END`) or load the row in app code and parse with the schema.
  - **Readers must reach for `.text` on each entry.** Code that previously rendered the entry directly (`${a}`) prints `[object Object]` after parse; the serializer is updated, any new UI is not exempt.
  - **Writers must take a `{ source, authorOperatorId? }` opts arg.** `dedupeAndAppendAntiPatterns` is the only current writer; any new helper that mutates this field must stamp the metadata so the migrated rows don't lose attribution.
- **Loyalty-program language is forbidden.** Points / rewards / tier / earn / badges / progress-bars in operator-facing or guest-facing surfaces violates the product principles at the top of this file. The voice critique system (THE-237/238) catches some drift; do not rely on it. Guests are *recognized*, not *enrolled*. If you write "this perk has been earned," the framing is wrong.
- **In-place mechanic edits without re-seed (THE-170).** `seed-venue` hard-fails on existing slugs and a full re-seed cascades through `venue_configs`/`mechanics`/`voice_corpus`/`voice_embeddings`/etc — destroying any Phase 5 voice corpus additions written by `ingest-response-review`. Set it to give a specific guest the fast, ungated demo experience — the agent runtime bypasses the TAC-212 approval policy gate IN FULL (including the comp regex backstop — see "Demo guest bypass" under "Approval policy gates") and skips the human-feel delay. - **`guests.context` writes are last-write-wins (TAC-296).** Two near-simultaneous inbounds from the same guest at the same venue both call `getGuestContext` against the same baseline, deep-merge their respective patches, and race on the final UPDATE. The second write clobbers the first; the losing patch's data is lost. **Why we accept this for v1:** pilot scale rarely produces concurrent inbounds from a single guest, and the lost write is recoverable (the agent re-captures on the next conversation). **When to revisit:** if pilot data shows the pattern, add optimistic locking via `guests.updated_at` precondition + retry-on-conflict, mirroring the TAC-264 partial-unique-index approach for `messages.review_state='pending'`. Don't ship that complexity until the failure mode is observed.
- **Reading `guests.context` directly via SQL or jsonb paths is brittle.** Always go through `GuestContextSchema.safeParse` (or `getGuestContext` from `lib/guests/context.ts`). The schema is permissive (unknown keys stripped, no `.strict()`), so future migrations can extend the shape without breaking older deploys — but raw `->` / `->>` paths in SQL or app code bypass `filterActiveLifeContext` (expired entries leak into the prompt) and the observations-truncation-to-10. If you need a SQL surface (e.g. a Command Center read-only viewer), call `getGuestContext` from the route handler and pass the parsed shape; don't render JSONB paths raw.
- **`contextUpdate` is a required schema field, inner shape is `.optional()`.** Per the TAC-212 precedent (Anthropic's structured-output validator is more reliable with explicit field presence), `GeneratedMessageSchema.contextUpdate` must be present on every emission — but both inner sub-fields (`structured`, `observation`) are `.optional()` so the no-op shape is `contextUpdate: {}`. The orchestrator's `isEmptyContextUpdate` check short-circuits before any DB hit. **Implication for tests / mocks:** any new fixture that constructs a `GenerateMessageResult` MUST include `contextUpdate: {}` or the typecheck fails. Existing fixtures (`stages.test.ts`, `schedule-and-send.test.ts`, etc.) were updated in the TAC-296 ship; new ones inherit the same requirement.
- **TAC-297 adds two more required-with-optional-inner schema fields: `commitment` and `arrivalCapture`.** Same Anthropic-structured-output reasoning as `contextUpdate` — both must be present on every emission, both have all-optional inner shapes (no-op shapes are `commitment: {}` and `arrivalCapture: {}`). Helpers `isEmptyCommitmentEmission` + `isEmptyArrivalCapture` short-circuit; `pendingFromEmission` returns null on no-op. **Implication for new fixtures / mocks:** every fresh `GenerateMessageResult` literal must carry `contextUpdate: {}, commitment: {}, arrivalCapture: {}` or tsc fails. Existing TAC-296 fixtures were extended in the TAC-297 ship.
- **`GeneratedMessageSchema` optional-field budget (TAC-300).** Anthropic's structured-output validator rejects tool schemas with more than 24 optional parameters across the entire nested tree ("Schemas contains too many optional parameters … limit: 24"). The CI guardrail in `lib/ai/schema-budget.test.ts` walks the schema's ZodOptional wrappers and fails CI if the count exceeds 22 (a 2-slot buffer below the cap so a single near-term field can land without a coordinated reshape). The TAC-296 → TAC-297 accumulation hit 31 and took production generation down for ~hours on 2026-05-30 — TAC-300 reshaped `GuestContextPatchSchema` (collapsed `home_base`/`workplace` to bare strings, dropped `pronouns`/`date_of_birth`/`observations` from the patch, dropped patch `captured_at`) to land at 20. **When adding a new agent-emission field:** a required boolean/string is free (Anthropic counts optionals only); a single optional leaf is 1 slot; an optional nested object is 1 slot for the object PLUS one for each optional leaf inside (`home_base: { neighborhood?, zip?, city?, address? }` was 5 slots, not 1). If the new field would push the count over 22, reshape first (collapse to a single optional string, fold into the `observation` free-text shortcut, or split into a separate tool call) BEFORE merging. The guardrail counter is OUR reconstruction of Anthropic's rule — it matched the validator's reported 31 on the pre-fix schema, but the real proof is the prod smoke test on every schema change.
